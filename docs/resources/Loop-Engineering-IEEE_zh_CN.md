# Loop Engineering：Anthropic 关于设计“为你的 Agent 发提示”的系统手册

一项关于“设计会自行运行的循环”的田野研究

摘要——过去两年里，一连串“XX Engineering”术语几乎紧跟模型发布节奏不断出现。本文考察其中最新的一个：Loop Engineering。这个术语于 2026 年 6 月由 Peter Steinberger、Boris Cherny 和 Addy Osmani 分别独立提出，其中 Osmani 首次以书面形式命名。与提示工程、上下文工程或 Harness Engineering 不同，Loop Engineering 不是教实践者如何把工作做得更好，而是让实践者不再处于“亲自做这项工作”的位置。本文给出其定义，将其放在 harness 之上的第四层，并把循环中的单轮拆解为五个动作：发现、交接、验证、持久化与调度，以及实现这些动作的六个组成部分。我们特别强调生成器/评估器分离：经验上，让一个 agent 给自己的输出打分时，它往往会夸奖自己；而调校一个独立、怀疑性的评估器，比让生成器对自己变得苛刻要容易得多。我们考察了三个正在实际运行的 loop，从单个工程师的晨间分诊，到 Stripe 每周合并 1300 多个机器撰写的 pull request 的企业级流水线，并归纳出四种会悄然累积的成本：验证债务、理解腐化、认知让渡与 token 爆炸。最后，我们给出构建第一个 loop 的具体配方。本文的核心主张是：loop 让生成几乎变成零成本，而判断成为稀缺资源；同一个 loop，由两个人构建，可能导向截然相反的结果。

关键词——Agentic AI，软件工程，自主代理，编码代理，生成器-评估器，调度，自动化。

## 一、Loop Engineering 到底是什么

提示工程课程还在持续售卖，上下文工程的墨迹还未干，Harness Engineering 也才刚刚被系统化写出来。现在，Loop Engineering 也来了。过去一年里，这些“XX Engineering”术语几乎随着模型发布一同出现，因此人们翻白眼的冲动是可以理解的。

但这个术语不一样。它不是关于如何把工作做得更好，而是关于把实践者彻底从“亲自做工作”的位置上移开。此前的术语都默认一个人坐在键盘前，逐行指挥 agent。Loop Engineering 删掉了这个前提。实践者不再位于循环内部，而是站在循环外部，负责构建这个循环。

## A. 一句话定义

给这个术语命名并把它系统写出来的人，是 Google Chrome 团队的工程师 Addy Osmani。他的定义很短：Loop Engineering，就是用系统取代自己成为那个给 agent 发提示的人。你不再逐行给 agent 喂提示，而是去设计一个会自动给它喂提示的系统。句子的重心在“取代你自己”。这是一种位置转变：从引擎本身，变成设计引擎的人。你写的，不再是给 agent 的一句句提示，而是一个会自动向 agent 发送这些话的东西。

## B. 一周之内，三个人同时点燃了引信

这个术语不是凭空冒出来的。2026 年 6 月的一周内，几组人几乎在同一时间撞见了同样的现象。导火索是 Peter Steinberger（OpenClaw 作者）的一条帖子，浏览量超过 800 万：人们不该再亲自提示编码 agent，而应该去设计“提示 agent 的 loop”。几乎同一时刻，Anthropic 的 Claude Code 负责人 Boris Cherny 也在说类似的话——他不再直接提示 Claude，而是让一些 loop 运行起来，由它们去提示 Claude、判断该做什么，而他的工作变成了写 loop。6 月 7 日，Osmani 在博客中以《Loop Engineering》为题把 Steinberger 和 Cherny 的说法整理出来，并在次日同步到了 Substack。一处点火，一处回响，一个名字，全都发生在同一周。

把这三种表述叠在一起，它们指向的是同一个动作：人们设计的对象，已经从 agent 的某个单一行为，转向驱动 agent 的整个系统。就像此前的 “vibe coding” 一样，这个词听起来也许粗糙，但它把一种工作方式变成了可讨论之物。

## C. 为什么这个术语恰好现在出现

值得追问的是：为什么三个彼此没有对过表的人，会在同一周抓住同一个词？答案是，周边工具已经悄悄跨过了一个阈值。编码 agent 已经足够可靠，能在无人盯守下完成非平凡任务；主流 harness 已经出现调度原语；单次 agent 运行的成本也已经下降到一个程度，以定时方式重复运行它不再显得浪费。当所有部件都已经到位时，把它们组合起来的那个动作就会同时对所有人变得显而易见。名称比实践晚了几个月：在人们给它命名为 loop engineering 之前，大家就已经在写 loop 了；就像团队先开始把写作 agent 与审查 agent 配对，然后生成器/评估器分离才有了名字。

这种“实践先行，命名滞后”的模式值得记住，因为它提示我们该去哪里寻找下一个术语。它不会来自一次模型发布，而会来自这样一个时刻：某项新能力变得足够便宜，以至于一种原本难以想象的组合，突然变成了日常惯例。

## D. 位于 Harness 之上的一层

这种转变可以归结为两句话。在旧世界里，人坐在那里逐行提示 agent；它做完一件事就停下，等待下一步。人类是循环内部的时钟，每一次滴答都必须由人来触发。在新世界里，人设计的是一个会自己滴答的东西——它按定时器运行，派生辅助 agent 去做工作，并把自己的结果再喂回自己。正如 Osmani 所说，Loop Engineering 位于 harness 的上一层：下面的 harness 负责武装一次 agent 运行，上面的 loop 则让它自己一遍遍地运行。变化的是身份：从操作 agent 的人，变成调度它的人。价值从“知道如何指挥”转移为“知道如何构建 loop，以及如何在 loop 里装入一个能说不的检查点”——而后者正是最难的部分，后文会详细解释。

## 二、从 Prompt 到 Context，再到 Loop

这些“XX Engineering”术语并不是互相替代的；它们是层层叠加的，每一层关注的范围都更大。表 I 给出了这四层结构。

每往上一层，关注的单位就扩大一个量级：从一句话，到一个窗口，到一次运行，最后到一个会自行运行的 loop。图 1 以堆栈形式展示了这四层，其中 loop 位于 harness 之上一层。

## A. 四层结构

提示是最底层，也是最为人熟知的一层。它关注的是该告诉模型什么：措辞、示例、角色、语气。它的边界是一轮交互。但它的问题在于，它默认人类每次都在场，亲手把提示交进去。

上下文层把问题从“我该怎么说”提升到“为了让模型解出问题，这个窗口里应该放什么”。它关心模型的整个视野：该检索什么、如何总结、该清除哪些过时信息。一个充满噪音的窗口，会浪费掉最好的提示。

Harness 关注的是 agent 在一次运行中需要携带什么：哪些工具、哪些动作、何时加载上下文、如何从失败中恢复、什么状态算完成。它武装一次运行；但它不会让这次运行自动重复。

Loop 则把“等待你来继续”这件事自动化掉。有了前三层之后，agent 可以干净利落地跑完一次，但随后就停住。Loop 给它装上定时器，让它按计划醒来，派生 sub-agent 并行工作，再把自己的输出作为下一轮的输入。

## B. “上一层”到底多了什么

把 harness 与 loop 区分开的，是三个动词。第一，**按时运行**：loop 会按计划醒来，不需要按按钮。第二，**派生帮手**：一个转起来的 loop 会拆出 sub-agent——一个起草修改，另一个专门负责挑毛病审查。第三，**自我回灌**：loop 这一轮产出的内容，会成为下一轮的输入；昨天的发现会被写进文件，今天早上它再读回来，接着往下做。正是这种跨会话的记忆，使它成为“loop”，而不是把同一个一次性任务重复很多次。

这种细粒度区分很重要，因为每一层失败的方式都不同，而那个能说“不”的检查点也必须安装在不同位置。糟糕的 prompt 会当场暴露；糟糕的 context 会体现为错误答案。但在 loop 层，系统会在你睡觉时运行，修改你从未看过的代码，还会把自己的错误继续喂回下一轮——而这个错误可能好几天都不被发现。层级越高，人离现场越远，错误堆积的时间就越长。这正是 Loop Engineering 真正的困难所在：从来不是把 loop 搭起来，而是在 loop 里面装进去一个能把它停下来的东西。

## C. 每一层失败的“爆炸半径”都不同

想象同一个底层 bug——一个 agent 错读了某个函数的返回值——在每一层里会怎样表现。在 prompt 层，它只会在一次交互里产生一个错误答案；人立刻看到，并重写提示。在 context 层，误读来自被加载进窗口的过时文档；人发现答案自信但错误，于是清理上下文。在 harness 层，agent 基于这个误读行动一次——例如编辑了一个文件——但运行会结束，diff 是可见的，人在任何东西发出去之前就会审查它。而在 loop 层，同样的误读会被写进状态文件，第二天早上又被读回来，当作既定事实继续往上堆。等有人终于去看时，这个错误假设已经成了承重结构。

这是 Loop Engineering 中最重要的直觉：**一个错误的代价，与它在被发现前存活的轮数成正比；而 loop，按其定义，就是一个用来最大化轮数的机器。** 后文里所有东西——评估器、人工检查点、预算上限——都是为了缩短错误与被发现之间的距离。

## 三、一个 Loop 的五个动作

“loop” 这个词很容易让人误解为原地空转。实际上，每一轮都在做具体事情：它找到值得做的工作，把它交给 agent，验证结果是否正确，保存状态，然后决定下一步。少了五个动作中的任何一个——发现、交接、验证、持久化、调度——这个 loop 就要么转不起来，要么只会原地打转。图 2 展示了这一轮如何喂养下一轮。

## A. 一个具体例子

Osmani 给自己做了一个晨间分诊 loop。每天早晨，一个自动化流程会自行启动。一个 triage skill 会读取前一天失败的 CI 测试、尚未关闭的 issue，以及最近的提交，并把结果写进一个 markdown 文件或 Linear 面板。对于每个值得采取行动的发现，它会打开一个隔离的 worktree；一个 sub-agent 起草修复，第二个 sub-agent 根据项目技能与测试进行审查。一个连接器会自动创建 pull request 并更新工单。任何它处理不了的内容都会进到一个待人工处理的收件箱里，并且会有一个状态文件保留下来，让第二天从上次停下的地方继续。整个流程不需要人工插手，但它也会在应该停下来等待人的地方停住。

## B. 这五个动作

**Discovery（发现）** 决定这一轮应该做什么。在这个例子里，triage skill 会读取 CI 失败、未关闭 issue 与最近提交。关键在于，让 agent 自己找到工作，而不是给它塞一个任务清单。更重要的是，这个自动化触发的是一个 skill——一种被永久化的知识——而不是贴在 cron 作业里、以后没人会维护的一大段说明文字。Discovery 决定了整个 loop 质量的上限：如果它挑出来的是没有价值的工作，那么剩下四个动作做得再漂亮，也不过是在为无意义的事服务。

**Handoff（交接）** 把任务从调度系统手里转交给真正执行工作的 agent。在这个例子中，每一个值得做的发现都会得到各自独立的 git worktree，这样多个 agent 就能在不同目录里改代码，而不会互相踩脚。任务切得越干净，后续的验证与合并就越容易。

**Verification（验证）** 是最容易偷懒、也最不能省的一步。第一个 sub-agent 起草修复后，第二个 sub-agent 会对它进行审查——用不同的指令，有时甚至用不同的模型。写代码的 agent 给自己作业打分时总会过于宽松；而专门负责找洞的 agent，能抓住第一个 agent 被自己说服而放过的问题。这就是那个“能说不的东西”。没有真正检查的 loop，不过是一个只会对自己点头的 agent。

**Persistence（持久化）** 把结果落到能穿越会话而存在的地方：通过连接器创建 PR、更新工单、把无法处理的内容放进收件箱，并用状态文件记录进度。Loop 的记忆不能只活在上下文窗口里；写进 markdown 或看板里的东西才不会忘。

## 四、六个组成部分：Loop 由什么构成

如果说动作描述的是单轮中发生了什么，那么组成部分描述的就是：要想让它转起来，手里必须有什么。这两者是对应的：发现依赖 skills，交接依赖 worktrees，验证依赖 sub-agents，持久化依赖 memory，调度依赖 automations。

**Automations（自动化）** 让 loop 能够自己动起来，挂在某个计划或触发器上。没有调度，你手里有的只是一次运行，不是 loop。自动化触发的应该是一个命名的 skill，而不是 cron 作业里一堵没人会再更新的说明文字。调度也不止一种形式——本地调度（机器必须保持开机）和云端调度（机器关掉也继续运行）。

**Worktrees** 是 git 自带的机制，可以在同一个仓库中开出多个独立工作目录。它的价值会随着并行度提升而放大：两个 agent 同时写同一个文件，和两个工程师同时改同几行代码一样麻烦。Worktree 把“虽然能并行但很乱”变成“既能并行又很干净”。

**Skills** 把项目知识永久写进一个文件（SKILL.md）中，这样 agent 就不必在每一轮重新推导上下文。Osmani 把 skills 抵消的成本称为意图债（intent debt）：也就是一遍又一遍解释“这个项目是什么、规则是什么、陷阱在哪里”的代价。Skill 可复用、可维护；一堵 prompt 文字则不行。

**Connectors（连接器）** 基于 MCP（Model Context Protocol），把 loop 接到外部世界——issue tracker、数据库、staging API、Slack。一个只能看见文件系统的 loop，只是个很小的 loop。连接器决定了 loop 的视野半径，而且为一个工具写好的连接器，往往可以直接挪到另一个工具上继续使用。

**Sub-agents** 把“写的人”和“评的人”分开。当同一个 agent 既当选手又当裁判时，裁判总会偏袒自己。反直觉的是：调一个独立裁判，让它变得挑剔，远比让生成器对自己的作品严苛更容易——这就是为什么一个 loop 要保留额外的 agent，而不是让同一个 agent 审计自己。

**Memory（记忆）** 是存在于单次对话之外的持久状态——一个 markdown 文件，或一个看板。上下文窗口一被清空，agent 立刻就什么也不记得了；如果一个 loop 想在今天接着昨天继续，它的记忆就必须落到磁盘上。agent 会忘，仓库不会。记忆不是上下文：上下文是这轮 agent 能看见的东西，会在刷新时被清掉；记忆则跨轮次、跨日持久存在。表 III 展示了六个部分与五个动作之间的映射。

六个部分都到位后，一个 loop 才有骨架：自动化让它动起来，worktree 防止它和自己打架，skill 让它不用重复解释背景，连接器让它看见外部世界，sub-agent 让它能纠正自己，memory 让它能记住之前发生的事。但把它搭起来只是开始：同样一套部件，由两个人去构建，最后可能长成完全相反的东西。

## 五、生成器与评估器

一个 loop 最难的部分，不是让 agent 跑起来，而是在里面放一个会说“不”的东西——而写代码的 agent 恰恰是最不可能说这句话的那个。

## A. 它总会夸自己

让一个 agent 评价自己刚刚产出的内容，它往往会非常自信地夸奖自己，即便人类一眼就能看出质量平平。Anthropic 工程师 Prithvi Rajasekaran 在构建长时间运行的应用时就观察到了这一点。这不是“够不够聪明”的问题，而是“给自己的作业打分”的问题。代码写出来时所处的上下文，已经塞满了“为什么要这么写”的理由，所以当 agent 回头看自己的输出时，它看到的不是结果本身，而是那条把自己一步步说服过来的自我论证链。在 loop 里，这个缺陷会被放大：如果每一轮“这够不够好”的判断都由刚刚写出它的 agent 自己做，那么它每一轮都会对自己点头，而且运行得越久，偏离真实质量就越远。

![image](https://cdn-mineru.openxlab.org.cn/result/2026-06-28/f2e7625e-b8c6-4089-b094-94ad37236d11/1f16b96ca85b3b07d3d955c19455c3591b3bd711d5df7a122d0bacf46706559c.jpg)

图 3. 生成器与评估器作为两个独立 agent。评估器不携带生成器的自我说服链，默认持怀疑态度，并通过“动手验证”而不是只读代码来做判断。

## B. 调一个怀疑者，而不是修一个谦虚作者

让生成器变得更会自我批评，效果通常很差。Rajasekaran 发现，调校一个独立的评估器，让它变得怀疑，比让生成器对自己的作品变得苛刻要容易得多。区别是结构性的，不是措辞问题：你无法要求作者跳出自己的视角，但你可以换上一个带着完全不同指令的另一个 agent，让它从头审视代码，不携带任何自我说服过程。这个思路借鉴自生成对抗网络（GAN）：一个网络负责构建，一个负责找错；只是这里被移植成了写代码的生成器与做审查的评估器。图 3 展示了这个结构形成的 loop。

## C. 评估器应该“动手”，而不只是“阅读”

仅仅把 agent 换掉还不够。如果评估器只是读代码，它判断的是“这看起来对不对”，而不是“它跑起来对不对”。在前端任务中，Rajasekaran 给评估器接上了 Playwright MCP，让它能像 QA 工程师一样打开页面、点击按钮、截图、检查 DOM。这样判断依据就从“这段 JSX 看起来没问题”转成了“我点了按钮，页面发生了跳转，这里有截图作证”。切换底层模型也有帮助：同一个模型即使用了新指令，也常常保留原有盲点。社区里常见的一种校准方式，是明确告诉评估器：**先假设这段代码是坏的，除非它被证明没问题。** 默认姿态应该是怀疑，而不是信任。

## D. 在产品中：把 `/goal` 用在停止条件上

Claude Code 把这种结构做成了一个原语：`/goal`。给 agent 一个条件，让它持续运行直到条件满足。一个具有代表性的评估器配置和停止条件如下所示。

```txt
# 评估器 agent（.claude/agents/reviewer.md）
角色：对抗式代码审查者。
默认：除非被证明没问题，否则假设这段代码是坏的。
不要表扬。只找失败点。
按以下顺序检查：
```

```txt
1. 它能运行吗？（执行，不要只读）
2. 跑测试：执行它们，并粘贴真实输出。
3. 作者跳过了哪些边界情况？
4. 行为是否符合工单要求？

使用 Playwright MCP：打开页面、点击、
截图、检查 DOM。判断行为，
而不是意图。

结论：只有所有检查都通过时才判定 PASS。
否则 REJECT，并列出每一个原因。

# 停止条件，由一个新的小模型来判断
/goal all tests in test/auth pass and the lint
step is clean
```

关键在于：每一轮结束后，会有一个小而快的模型检查这个条件是否已经满足；如果没有，系统就继续再跑一轮，而不是把控制权交还出来。完成与否是由一个“新鲜模型”决定的，而不是由执行工作的那个模型自己决定。这就是 maker-checker 原则——它在银行业里已经存在几十年，比如输入大额转账的人和复核的人必须不同——现在被应用到了停止条件的判断上。（Codex 也可以通过自动化与 agent 配置实现同样能力；不要把 `/goal` 和 `/loop` 混淆，后者只是按时间间隔重复运行。）

一个 loop 的下限，由它的评估器决定。生成器的水平决定 loop 能产出什么；评估器的水平决定它**不会**产出什么。把生成与判断在结构上分离；把评估器调成一个怀疑者；让它通过实际操作而非阅读来验证；再把最终裁决交给一个新鲜模型——这四步，构成了一个 loop 说“不”的能力。

## 六、Loop 出错的五种方式

在讨论有效运行的 loop 之前，先把它们失败的方式梳理出来更有价值，因为失败比成功更有启发性，也更常见。下面每一种反模式，都对应单轮五个动作中的某一步被跳过或做坏了。

## A. 点头循环（跳过验证）

这是最常见的失败方式。Loop 跑起来了，agent 写了代码，然后同一个 agent 宣布它没问题。因为没有独立检查，每一轮都会产出“自我批准”的结果，loop 便以机器速度积累起看起来很合理、实际上有问题的错误。症状是：一个 loop 连续跑了几百轮，却从未对自己说过一次“不”——对于任何真实工作负载来说，这在统计上都不可能，因此只能说明根本不存在真正的检查。解决方式就是上一节提到的生成器/评估器分离。

## B. 失忆循环（跳过持久化）

Loop 找到了值得做的工作，也做了，但随后它忘了自己做过这件事，因为结果只活在一个被清空的上下文窗口里。下一轮它会重新发现同一项工作，或者更糟，再做一遍并和第一次的结果发生冲突。症状是：loop 没有任何累积进展；每天早晨它都从同一个地方重新开始。解决方案是磁盘上的状态文件——agent 会忘，但仓库不会。

图 4. 每一种反模式，都是五个动作中某一步被跳过。五种失败与单轮循环的五个动作一一对应。

## C. 手动循环（跳过调度）

一个 loop 即使做好了前面四个动作，只要没有自动化，它也不是真正的 loop；它只是一个需要人手动运行的脚本，而人总有一天会忘了再运行它。它在构建当天看起来很惊艳，但在注意力转移的那天就会悄然停摆。症状是：这个 loop 的最后一次运行，正是演示它的那一天。解决方案是真正的触发器——定时器或事件——而不是依赖人类记得去执行。

## D. 盲目循环（跳过发现）

人类仍然每天早晨亲手把工作交给 loop——“修这三个 bug”——所以它自动化了“做”，却没有自动化“找”。这节省的时间没有看起来那么多，因为挑选该做什么，本身往往才是昂贵的部分。症状是：人类依然把早晨花在决定 loop 今天该干什么上。解决办法是把发现逻辑写进 skill，让 loop 自己浮现值得做的工作。

## E. 缠结循环（跳过交接）

Loop 同时并行运行多个 agent，但它们都在同一个工作目录里改代码，于是修改相互碰撞，合并一团乱麻，无人能理清。这个问题只会在并行场景下暴露：单 agent loop 看起来一切正常，而第一个“五个 agent 同时跑”的早晨就会出事。解决方案是：每个任务一个隔离 worktree。图 4 展示了五种反模式与它们违反的动作之间的对应关系。

这五种失败不是彼此独立的。缺失验证的 loop 往往也缺失持久化，因为对一个检查马虎的团队，通常也会对其他检查马虎。在实践中，它们经常成簇出现：纪律严明的 loop 会把五个动作都装上，而仓促搭起来的 loop 通常只装了“发现”和“交接”——也就是那两个会产生可见输出的部分——却跳过了另外三个负责安全的部分。下一节会展示三个把五个动作都装全了的 loop。

## 七、在你睡觉时运行的 Loop：三个真实案例

下面三个公开案例在规模上差异极大，但骨架相同：一个触发器负责启动，一组约束让它不至于跑偏，而终点处保留一个人工检查点。“在你睡觉时运行”从来不是靠模型有多强，而是靠这副骨架有多扎实。

## A. 一位工程师的早晨

Osmani 的晨间分诊 loop 已经在第三节中拆解过。这里值得再次强调的唯一细节是：自动化调用的是一个 skill，而不是塞进调度器里一大段以后没人再会更新的指令。这个案例展示了一个人可以运行的 loop 长什么样——一个人、一台机器、每天早上自动完成那些琐碎但必要的工作。

## B. Stripe 的 Minions：每周 1300 个 PR

如果要看企业级规模的案例，最值得研究的是 Stripe 的 Minions：每周合并 1300 多个 pull request，而且没有一行是手写的。这个案例出自 Stripe 工程师 Steve Kaliski 在播客 *How I AI* 中的介绍。触发器本身很轻——在 Slack 里 `@` 一下机器人，或者加一个 emoji reaction 即可。真正让它可靠的，是模型醒来之前那一大段确定性编排：一个确定性的 orchestrator 会先组装上下文，扫描链接、拉取 Jira、查找文档，并用 Sourcegraph 加 MCP 来定位相关代码。让 LLM 自己去找上下文，是整个系统中最难控制的一部分，因此凡是能用硬编码规则解决的事，就都被拿出模型之外。**任何可以用确定性逻辑解决的事，都不应该交给概率模型；而你把这条线画在哪里，决定了这个 loop 是否可靠。**

最反直觉的一点是：Minions 并不是建立在一个更强的模型之上。它是开源工具 Goose 的一个分支，而它的核心论断是：可靠性来自约束质量，而不是模型大小。它的架构像图 5 所画的那样，在确定性关卡与创造性 LLM 步骤之间交错：agent 写代码，随后一个硬编码流水线运行 linter，agent 无法跳过它；agent 再修 lint，随后另一个硬编码步骤执行 commit。沙箱环境是运行在 EC2 上的 Devbox，并采用 “cattle not pets” 的思路管理：每个环境都可以随时替换，因此一千多个 agent 可以同时运行而互不干扰。值得注意的是，那 1300 个 PR 仍然会由人类审查——人类并没有离场，只是换了张桌子，从写代码的人变成了审查代码的人。

## C. “在你睡觉时运行”真正依赖什么

本地 `/loop` 与桌面计划任务都要求机器保持开机；一旦你把机器关掉，loop 就会停止。若希望机器关掉后仍能运行，正确答案是 Cloud Routines 或 GitHub Actions 的计划触发器。表 IV 对这些选项进行了对比。如果你希望它高频运行，而且能访问本地文件，那么用本地 `/loop`，代价是机器必须开着。如果你希望它脱离本地状态独立运行，那就上云，代价是最小间隔通常是一小时，而且每次都是一个新的 clone。没有任何一种调度器能兼顾一切。

## D. 在实践中如何选择调度器

在本地与云端调度之间做选择，不是审美问题，而是由一个问题机械地决定：**这个 loop 的工作是否绑在本地机器上，还是可以脱离它？** 两个具体场景能把规则说明白。假设一个 loop 必须每分钟检查一次本地开发服务器——这项工作只能本地运行，因为云端看不见你笔记本上的进程，而且云端间隔也无法低于一小时。反过来，假设一个 loop 需要在凌晨三点扫描仓库里的开放 issue，并在适当时自动创建 PR——这类工作就绝不该绑在一台笔记本上，因为笔记本会合盖、会断电、会被带走。对后者来说，云端计划任务或 CI 的 schedule trigger 才是正确答案：运行在始终醒着的机器上，而此时人类已经睡着。

应当避免的认知扭曲，是把“本地反复重跑”当成“睡觉时运行”的全部含义。本地反复重跑意味着“我人在这里时，多跑几轮”；云端调度意味着“即使我不在，这东西也会继续跑”。这是两种不同的能力，把它们混为一谈，只会让人合上电脑盖后失望地发现：自己以为已经自治的 loop，悄悄停了。更诚实的表述是：本地调度换来更高频率与访问本地文件的能力，但代价是机器必须保持开机；云端调度换来真正自治，但代价是更粗的时间粒度与每次全新 clone。没有哪种调度器能包打天下，成熟的 loop 往往两者并用——本地用于紧密的内部检查，云端用于夜间全局巡检。

## E. 同一种能力，两条工具链

本文中的命令来自 Claude Code，但这些能力并不属于某个产品。Codex 以不同名字提供同样的五脏六腑，而一个为一侧编写的连接器，往往也可以移动到另一侧继续使用。表 V 把它们逐项对齐，避免读者把某个命令名误当作本体。教训是：Loop Engineering 是一组能力，而不是一个产品。它包括调度、运行直到条件满足、并行隔离、sub-agent、外部连接与显式 skill 调用。无论团队使用什么工具链，真正该问的是这六种能力是否都在，而不是哪家产品的哪个命令提供了它。

## 八、代价：四笔不会自己清零的账

一个会自己运行的 loop，同时也是一个会自己犯错的 loop。它运行得越轻快，犯错就越安静。会累积四种成本，而且在 loop 运行期间，它们都不会主动拉响警报。

**验证债务（verification debt）**。每一个被打开并合并的 PR 都节省了时间，但这些节省下来的时间会变成尚未验证的输出，迟早要偿还。问题藏在测试覆盖不到的地方，藏在“能跑”和“正确”之间的缝里，并持续积累，直到某个发布日一起爆炸。防线是一个独立的评估器——与执行工作的 agent 分离。

**理解腐化（comprehension rot）**。Loop 越快地交付你没亲手写过的代码，你对代码库的实际理解与它真实状态之间的差距就越大。读代码总比写代码无聊，而 loop 把“写”拿走了；于是代码库不断增长，你脑中的地图却停滞不前。它不会发出警报，直到某个 bug 钻进了你从未看过的角落。防线是定期阅读 loop 的输出，并强迫自己解释其中一部分改动；解释不出来，就说明你的地图需要更新了。

**认知让渡（cognitive surrender）**。当 loop 自己跑起来后，人很容易不再主动形成判断，而只是接受它交回来的结果。这是前两种问题在态度层面的版本：不是“没时间”，而是“懒得再管”。Loop 越可靠，越容易把判断力外包出去。防线只有一句话：**loop 可以执行，但它不能替你决定。** 至少，你必须仍有能力说出“这是错的”。

**Token 爆炸（token blowout）**。这是唯一直接体现在账单上的成本，而且事先很难估算：loop 会孵化辅助 agent、会重试、会一轮又一轮地运行，所以一个 bug 就可能空转一整夜，最终留下的是一张陌生账单，而不是修好的代码。防线是在投入运行之前就设置硬上限——每轮预算、每日预算、最大重试次数——这样一个空转 bug 就烧不掉整晚配额。

这四种成本有一个共同特征：**它们在 loop 运行时都是沉默的。** Loop Engineering 最迷人的地方，在于它让一个人做出一个团队的工作量；但最危险的地方也恰恰在这里，因为一个团队至少会彼此争论，而一个人加上一堆 loop，很容易变成一个没有人争辩的回音室。

图 6. 四种成本会互相强化。未经验证的输出侵蚀理解，理解的衰退促使认知让渡，认知让渡让 loop 在更少监督下运行更久、花费更多，进而产生更多未经验证的输出。

## A. 一个“债务复利”的具体例子

设想一个 loop 一夜之间开出了二十个 pull request，而且全部测试都是绿色。表面上，这似乎是一次胜利。但假设其中有三个 PR 带着测试未覆盖到的细微错误。没有独立评估器，这三个 PR 会被合并——这就是验证债务。因为人类没有阅读这二十个 PR 就全部合并，他们对代码库的心智模型现在落后了二十次改动——这就是理解腐化。因为这个 loop 跑得过于顺滑，人类第二天早上干脆不再阅读新一批输出——这就是认知让渡。而由于 loop 一夜之间不断派生 helper 并自由重试，账单变成了原估算的三倍——这就是 token 爆炸。那三个被埋起来的错误，此时已经坐落在一个人类不再完全理解的代码库里，外面再由一个已经不再审视它的人把守；它们最终很可能只会以一次线上事故的形式被发现。

这个例子的重点在于：四种成本不是四个互相独立的风险，而是**同一个失败的四张面孔**。它们彼此强化：未验证输出越多，人类理解越少；理解越少，让渡越多；让渡越多，loop 无人看管地跑得越久，账单越大。图 6 展示了这种正反馈回路。防御这四者的方式其实相同：让人类始终保有说“不”的能力，并在系统中装入一个不需要等人醒着才能运行的检查点。

## 九、继续做工程师，而不只是那个按下开始按钮的人

同一个 loop，由两个人来建，可能走向完全相反的结局——差别不在 loop，而在人。正如 Osmani 所说，两个人可以构建同一个 loop，却得到截然相反的结果。一个人用 loop 去加速自己已经理解透了的事情：他读代码，掌握方向，loop 只是放大了他本来就具备的判断力。另一个人则用同样的 loop 来逃避理解本身。六个月之后，前者变得更强，后者则成了一个自己都看不懂的机器的看门人。

Loop 不是一种“质量由工具自身决定”的工具。它强大到会原样放大构建者带进去的一切：你带进去理解，它就放大理解；你带进去懒惰，它就放大懒惰。它是一个忠实的乘号，而它乘上的，是人本身。

Loop 让生成极其便宜——代码、计划、PR、修复，都几乎免费。真正仍然稀缺的是判断力：知道哪个方案是对的，知道哪一行该被拦下，知道哪份输出虽然能跑却在根上错了。Loop 可以生成一百个选项，但它不能真正做出选择；或者更准确地说，它依据的是“看起来合理”而不是“事实上正确”来选择，而这两者之间的差距，正是工程师存在的理由。因此，Loop Engineering 不是贬低判断，而是把所有不需要判断的部分都剥离掉，最终只剩下判断本身。

Loop 会忠实执行交给它的逻辑，但它并不理解你为什么要构建它、你真正想要什么，或者哪些地方你宁愿亲自盯着。这些边界——人工检查点放在哪里，哪里让它自己跑——不是 loop 自己能读出来的；它们只存在于构建者脑中，必须一个个写进去。你设计它时带着什么心态，它最终就会长成什么形状。一个抱着“赶紧把我解放出来”心态构建的 loop，和一个抱着“我仍要做工程师”心态构建的 loop，代码上可能有 90% 一样；差别也许只在一两个检查点，而正是这些检查点决定了六个月后，到底是你站在 loop 之上，还是被它掏空。Osmani 的结语值得记住：构建 loop，但要像一个打算继续做工程师的人那样去构建，而不是像一个只想按下“开始”按钮的人那样。

## 十、判断力的经济学

上一节提出了一个值得单独展开的论断：loop 让生成变得便宜，而让判断保持稀缺。这不是一句口号，而是一个具有组织后果的经济学观察。

## A. 什么变得充裕

当一种资源变得充裕，它的价格就会下降，而围绕它组织起来的活动也会重组。Loop 让代码、计划、修复和 pull request 变得充裕——一个拥有良好 loop 的工程师，能够产出相当于一个小团队的输出。那些原本占据工程师一天时间的活动：敲代码、样板劳动、机械式重构，成本都会塌陷到接近零。一个很自然的第一反应，是认为这会让工程师变得不再值钱。事实恰好相反，但前提是工程师仍握着那个稀缺之物。

## B. 什么仍然稀缺

稀缺资源，是决定从大量输出中保留哪一个的判断力。Loop 可以生成一百种候选实现；它无法告诉你哪个是真的对，只能告诉你哪个“看起来合理”，而“看起来合理”和“真正正确”之间的间隙，正是工程存在之处。随着生成趋近免费，工程师的全部价值会集中到这个间隙之中。剩下的工作，是纯粹的判断，被从原先包裹它的机械劳动中提炼出来。

这带来一个令人不适的含义。如果一个工程师的价值主要来自机械劳动——打字快、API 背得多、愿意磨样板代码——那么这部分价值会消失，因为 loop 可以免费做掉这一切。而如果一个工程师的价值来自判断，那么这部分价值会被放大，因为 loop 会把他的好决策忠实执行一百遍。**同一个工具会拉大这两类工程师之间的差距。** 它不会平等抬升每一个人；它只会放大每个人原本带来的东西。

## C. 放大器两边都割人

因为 loop 是判断力的放大器，所以判断失误同样会被放大。在旧世界里，一个坏决策的代价只是一小段手写的错误代码，爆炸半径有限，而且速度慢到足以在途中被发现。而在新世界里，一个坏决策会被机器大规模、忠实地执行一百次，而且它不会停下来问一句“这真的对吗”。Loop 移除了过去那档“慢速齿轮”，那档齿轮曾经在很多时候救过工程师。你不能再指望流程慢到足以在中途发现错误，因为流程已经没有慢速档了。这提高了对 loop 无法替代的那件事的要求，也正因为如此，“继续做工程师”并不是情怀，而是运行层面的必要条件。

## 十一、运行纪律

如果判断是稀缺资源，那么实际问题就是如何把它花在刀刃上。结合上文案例与成本，有三条纪律值得作为长期实践明确写下来。

## A. 永远抽样阅读

防止理解腐化的办法，不是把 loop 产出的所有东西都读完——那会违背使用它的初衷——而是**每天读一个有代表性的样本**，并强迫自己解释样本中的每一项改动：它做了什么，为什么要这样做。解释不出来，恰恰就是一个精确的信号，说明你脑中的地图已经落后于代码库。从一个安静早晨里抽样的 PR 中发现这一点，远比在一次糟糕日子的线上事故里发现要便宜得多。样本不需要很大；它需要的是规律，而且必须真的被看过。

## B. 上线前先设上限

防止 token 爆炸的办法，是在 loop 第一次无人值守运行之前，就设置硬性天花板，而不是等第一张令人惊讶的账单出来以后再补。单次运行预算、每日预算和最大重试次数，三者合起来才能确保一个空转整夜的 bug 不会吃掉全部配额。这些数字首先不是为了省钱；它们是断路器，用来把一个开放式风险变成边界明确的风险。没有上限的 loop，本质上是在把它的花钱权委托给自己的 bug。

## C. 至少留一扇门开着

防止认知让渡的办法，是结构性的，而不只是态度上的。至少在 loop 里保留一个会为人暂停的检查点——不是因为人类每次都要介入，而是因为这个暂停点的存在，能让人类始终处在“随时可以介入”的位置上。把所有门都焊死、赌自己永远不需要进去的工程师，会在真正需要进去的那天发现，自己已经没有钥匙了。始终留着一扇门开的工程师，则随时能走进去看看 loop 到底在干什么。两个 loop 在代码上可能只差一个检查点，但六个月后，到底谁掌控局面，差别会大得惊人。

## 十二、今天就搭起你的第一个 Loop

Stripe 的流水线是终点，不是起点。第一个 loop 应该小到几乎不像一个系统——一个会按定时器检查某件事的小玩意。

第一步：运行一个 `/loop`。它在 Claude Code v2.1.72 后可用，会按时间间隔重跑同一任务。它是会话作用域的，循环任务七天后过期，并且运行在本地机器上；机器一关，它就停。

```shell
/loop 5m check the deploy # 固定：每 5 分钟一次
/loop check the deploy # agent 自己掌握节奏
/loop # 运行 .claude/loop.md
```

第二步：读取 CI 和 issue，先做分诊。重复跑一行命令并不构成 loop。你要给它一个提示，让它每天早晨看三样东西，然后列出哪些值得处理。被调度的自动发现，才是 loop 的入门形态。发现逻辑应该放在 skill 里，而不是调度配置里。

```txt
. claude/skills/morning-triage/SKILL.md
NAME: morning-triage
WHEN: 每天早晨由自动化调用。

READ:
- 昨天以来失败的 CI 运行
- 最近 24 小时新开的 issue
- 自上次运行以来合并的 commit

JUDGE: 对每一项判断，它现在是否值得处理？
跳过噪音，只保留可执行的发现。

OUTPUT: 将发现与状态写入
./state/triage.md（每个发现一行）。
```

第三步：加一个状态文件。不要把结果留在聊天窗口里。把每一项发现、以及它目前处理到哪一步，全部写进一个 markdown 文件（或 Linear 面板）里。agent 会忘，仓库不会。

```markdown
# ./state/triage.md（loop 的记忆）
| finding | source | status |
| --- | --- | --- |
| auth test flaky | CI #4821 | fixing |
| null deref | issue 92 | PR open |
| stale dep | commit a3 | inbox |
```

第四步：加一个评估器。这一步最关键，也最容易被跳过。Claude Code 的 `/goal`（自 v2.1.139 起）会持续运行直到满足某个条件，并用另一个模型判断该条件是否已经成立。

```txt
/goal all tests in test/auth pass and the lint step is clean
```

第五步：为并行加上 worktree。使用 `--worktree`（或 `-w`）为每个后台 agent 打开一个独立 worktree，这样它们就不会互相踩。

```shell
# 每个发现一个隔离 worktree
claude --worktree fix/auth-test "draft the fix"
claude --worktree fix/null-deref "draft the fix"
```

在表 VI 的六个要素中，前两个决定 loop 能不能跑起来；后四个决定它跑起来后会不会惹出麻烦。新手最常见的情况，是只把前两个装上就上线，结果得到一个谁都不盯、谁也拦不住、只会对自己点头的 loop。结论很简单：第一个 loop 宁可小一点，也要把“能说不的检查”和人工审查点真正装好。

## A. 一个完整且带注释的“第一个 Loop”

为了让检查表更具体，下面给出一个最小但完整的 loop，它把这六个要素全部装上了。它小到可以一口气读完，但已经具备一个真实 loop 所需要的所有器官，只是规模更小。

```yaml
# 1. 调度 —— 一个真正的触发器
# (.github/workflows/triage.yml)
on:
    schedule:
    - cron: '0 6 * * *' # 每天 06:00，云端运行

# 2. 发现 —— skill，而不是一堵文本墙
# 由 workflow 调用：
run: claude --skill morning-triage

# 3. 持久化 —— 把状态落到磁盘
# skill 会写入 ./state/triage.md
# 并把它提交回仓库

# 4. 交接 —— 每个发现一个 worktree
for finding in $(parse ./state/triage.md); do
claude --worktree "fix/$finding" \
--goal "tests pass and lint is clean" \
"draft a fix for $finding"
done

# 5. 验证 —— 一个新的模型来裁决
# /goal 的停止检查会在每一轮后运行；
# 另一个 reviewer agent 继续挑毛病

# 6. 人工审查 —— 永远敞开的门
# 只创建 PR，从不自动合并；
# 任何不确定内容都落到 ./inbox/
```

从上到下读，这六处编号注释正好对应检查表里的六个元素，而且每个都只用两三行就实现了。cron 那一行是调度；skill 调用是发现；被提交回仓库的状态文件是持久化；每个发现一个 worktree 是交接；`/goal` 的停止检查加 reviewer 是验证；“只开 PR、绝不自动合并”以及收件箱，则是人工审查点。一个把这六项都装上的 loop，即使很小，也是真正的 loop。缺任何一项，它都只是第六节那五种失败中的某一种换了件外套。

## B. 如何安全地把 Loop 扩大

一旦这个最小 loop 跑起来，诱惑马上就来了——更多发现、更多并行 agent、更短间隔。安全的扩张顺序，是**把并行放在最后加**，先证明检查机制可靠。应当先扩大它能发现什么，再扩大它能并行做多少；先证明评估器真的能抓住错误，再信任它去给很多 agent 同时做闸门。Stripe 的案例是这条路径的终点，而不是入口：它的可靠性来自多年对确定性关卡的加固，而不是一开始就搞得很大。Loop 只有先证明自己能拦住一个坏 agent，才算赢得了跑更多 agent 的资格。

其余的，不在本文里，在终端里。

## 十三、附录 A：带注释的分诊 Skill

发现动作依赖的是一个 skill，而不是一堵说明文字，因为 skill 可以维护、可以复用，而粘贴进调度器里的 prompt 只会慢慢腐烂。下面是前文多次提到的 `morning-triage` skill 的更完整版本，并加上注释，展示各部分如何服务于不同动作。

```markdown
# .claude/skills/morning-triage/SKILL.md
---
name: morning-triage
trigger: 由每日自动化调用
---
##
## 读取（DISCOVERY 的输入）
- 上次运行以来失败的 CI
- 最近 24 小时新开的 issue
- 昨天以来合并的 commit
- 上一个 ./state/triage.md

##
## 判断（决定上限的部分）
对每个候选项，判断：
    - 它现在可执行，还是噪音？
    - 它是否阻塞发布？→ priority
    - 它是否已经被跟踪？→ skip
只保留今天值得开 worktree 的事项。

##
## 写入（PERSISTENCE 的输出）
追加到 ./state/triage.md：
    | finding | source | priority | status |
提交该文件，以便明天继续读取。

##
## 交接（准备 HANDOFF）
对每个保留项，输出一行任务：
    worktree=fix/<slug> goal=<stop-condition>

##
## 停止（你为自己保留的边界）
绝不合并。绝不删除。任何你没有足够把握的内容，都放进 ./inbox/
留给人工处理，而不是直接进入 PR。
```

这个 skill 的六个标题里，有五个对应五个动作；第六个标题“Stop（停止）”，则是构建者把边界写进去的地方——那些 loop 自己推不出来、必须由人类决定的边界。Loop 会忠实执行 skill 里写下的一切，也会忠实忽略你没有写进去的东西，因此 “Stop” 这一节不是样板文本——它是工程师把控制权永久固化下来的唯一位置。省掉它，loop 就会带着自己并没有赚来的自信去自动合并。

## 十四、附录 B：术语表

## 十五、综合：这份手册最终归结为什么

跨越前面九个章节，整篇文章的论证主干只有一条。Loop Engineering 是一个从 prompt 开始、经由 context 与 harness 逐层上升而来的第四层；它与下面三层的区别在于，它把人类从“亲自做工作”的位置上移开。Loop 的单轮由五个动作构成——发现、交接、验证、持久化、调度——并由六个组成部分实现；而 loop 的各种失败，本质上只是这些动作被跳过。最难的动作是验证，因为 agent 在给自己的工作评分时总会夸奖自己，而可靠的补救措施是结构性的：一个独立的评估器，默认怀疑，通过行动而非阅读来验证，并由一个新的模型依据显式停止条件来判断是否通过。Loop 已经在实践中运行，从一个工程师的早晨，到一个企业每周合并上千个机器撰写 PR 的流水线；而真正让它们可靠的，是约束的质量，而不是模型的大小。它们会累积四种沉默的债务——验证债务、理解腐化、认知让渡与 token 爆炸——并互相强化，最终一起到期。而因为 loop 会放大构建者带进去的东西，同一个 loop 由两个人来建，最终会因为一两个检查点的差异而走向完全不同的结果。

真正该带走的一句话，是这个领域在一周内共同收敛出的那句：**不要再亲自去提示 agent，而要去设计那个负责提示 agent 的系统——但设计它时，要像一个打算继续做工程师的人，而不是一个只想按下开始按钮的人。** 本文里所有技术细节，都是为了服务这个姿态。评估器、状态文件、预算上限、那扇始终开着的门：它们每一样，都是让人类保有向一个高速说“是”的机器说“不”的能力的手段。Loop 是这一代软件实践中最强大的工具，正因为它是构建者最忠实的放大器；而一个忠实的放大器，价值有多大、危险有多大，完全取决于你喂给它的判断力。

## A. 第一个月的实践札记

对于开始采用 loop 的团队，有几条实践观察反复出现，值得明确说出来。第一，能活下来的 loop，是那个逐步赢得信任的小 loop，而不是那个一开始就要求别人无条件信任的宏大系统；先从一条发现开始，把它端到端跑通，只有在检查机制抓到过真实错误之后再逐步放大。第二，工程精力真正该投入的地方是评估器——强生成器配弱裁判，只会产生自信的垃圾；普通生成器配尖锐裁判，才会产生缓慢但可靠、能够复利的进展，而后者才值得累积。第三，人工审查点不是临时脚手架，等 loop “足够可信”以后就可以拆掉；它恰恰是保证 loop 持续可信的永久性特征，而它被拆掉的那一天，理解腐化就正式开始了。第四，预算上限应当建立在“迟早会有东西整夜空转”的假设上，因为这件事最终一定会发生，而上限的作用，就是把它从“账单上的一行惊悚数字”降级成“日志里的一个奇怪现象”。

这些观察单独看都不令人惊讶。真正让团队惊讶的是：一个“似乎就是能跑”的 loop 所带来的愉悦体验，会多快地腐蚀掉那些让它原本可靠的纪律；而且这种腐蚀在某个早晨之前都是看不见的。归根结底，这份手册并不是在教人怎么搭 loop——那部分如今真的已经很容易了——而是在提醒人如何继续成为那种工程师：无论哪天早晨醒来，都还能回答“这个 loop 刚刚做的那件事，究竟是不是真的对”。
